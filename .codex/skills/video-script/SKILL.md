# Video Script — Генерация сценария для YouTube

## Overview

Шаг 0 в контент-пайплайне. Пользователь приходит с темой или идеей → в диалоге уточняем концепцию → генерируем сценарий по формату Tone of Voice → после утверждения выдаём телесуфлёр-текст и DOCX. Если в сценарии нужны таблицы, схемы, дашборды, графики, KPI-блоки или сравнения, они сразу генерируются через встроенный GPT Image 2 (`gpt-image-2`) и вставляются в DOCX как embedded images, а не как текстовые таблицы.

**Важно про изображения:** для visual inserts не нужен `OPENAI_API_KEY` из `.env`. Использовать встроенную image-generation capability Codex / GPT Image 2. Не проверять наличие `OPENAI_API_KEY` и не считать его отсутствие блокером для генерации embedded visual images. Не заменять GPT Image 2 локальными Pillow/HTML/SVG-слайдами только из-за отсутствия ключа.

**Pipeline:**
```
[Шаг 0] Тема + дата/время публикации → Сценарий → Телесуфлёр (DOCX с зашитой датой в имени и embedded visual images, без служебных video-only маркеров) → мини-гайд следующих шагов SOP
[Шаг 1] Запись видео (пользователь)
[Шаг 2] Google Drive (имя YYYY-MM-DD-HHMM-Русский заголовок.mp4) → n8n (YouTube scheduled + thumbnail + уведомление)
[Шаг 3] /publish-from-script <youtube_url> → читает тот же DOCX → готовит WordPress/Дзен/Telegram контент → ставит в очередь Google Sheets
[Шаг 4] n8n Publisher Trigger (5 мин) → публикует в Telegram канал и блог в назначенное время
```

## Режимы запуска

**Режим диалога (по умолчанию):** пользователь даёт тему → обсуждаем угол подачи, ключевой кейс, цифры → генерируем сценарий.

**Express mode:** пользователь даёт тему + контекст в одном сообщении (например: "сценарий про бутылочное горлышко в закупках, кейс — металлургический завод, экономия 12 млн") → сразу генерируем сценарий без дополнительных вопросов.

## Workflow

### Step 1: Ввод и уточнение

1. Прочитать `tone-of-voice.md` из директории проекта, НЕ спрашивать у пользователя.
2. Получить тему от пользователя
3. **Получить дату+время публикации** (обязательно).
   - В диалоговом режиме — спросить: «Когда публикуем? Формат: YYYY-MM-DD HH:MM (MSK)».
   - В express mode — пользователь может указать в одном сообщении («сценарий про X, публикация 2026-05-01 14:00»).
   - **Валидация:** формат строго `YYYY-MM-DD HH:MM`, время не в прошлом (минимум now+15 мин). При ошибке — переспросить.
   - Сохранить как `pub_date` (YYYY-MM-DD) и `pub_time` (HHMM, без двоеточия — для имени файла).

**Если тема без деталей** (только заголовок/направление), задать 2-3 вопроса:
- Какой конкретный кейс или ситуацию разберём? (пример из практики, с цифрами)
- Какая главная боль аудитории в этой теме?
- Есть ли конкретные цифры/суммы для примера?

**Если тема с деталями** — сразу к Step 2.

**Не задавать больше 3 вопросов.** Недостающие детали — додумать самостоятельно в стиле Веселкова (инженерный подход, реальные цифры из нефтегаза/закупок/производства).

### Step 2: Генерация сценария

Структура сценария (из tone-of-voice.md, формат YouTube/Rutube):

```
[ХУК — 0:00] (15-20 сек)
Провокационное начало: угроза потери денег, замена ИИ, обман в отрасли.
Конкретная цифра или факт, который бьёт по больному.

[ТЕЗИС — 0:20] (20-25 сек)
Что покажем в этом видео. Обещание конкретного результата с цифрой.
"Сегодня я покажу схему, как..."

[ТЕОРИЯ — 0:45] (≤2 мин)
Краткая подача концепции/принципа.
Метафоры из производства. Без академизма — только практика.
Максимум 2 минуты, иначе зритель уходит.

[КЕЙС — 2:45] (4-6 мин)
Разбор конкретного примера с цифрами.
Кто, что, сколько, за какой срок.
Пошаговый алгоритм: что сделал → что получил.

[РЕЗЮМЕ — 8:00] (30-60 сек)
Жёсткое резюме без долгих прощаний.
3 ключевых вывода пунктами.
CTA: "Ссылка в описании", "Смотри следующий выпуск".
```

**Хронометраж:** 8-12 минут итого. Указывать примерные тайм-коды.

**Стиль текста:**
- От первого лица ("я")
- Императивное "ты" к зрителю
- Метафоры из производства/инженерии
- Формула "Проблема → Решение → Деньги"
- Конкретные цифры: суммы в рублях, сроки, проценты
- Капслок для акцентов в ключевых словах
- Короткие предложения, рубленый ритм
- Без психологии и мотивации — только инженерия процессов

### Step 3: Утверждение

Показать сценарий пользователю в компактном формате:

```
Сценарий: [НАЗВАНИЕ]
Хронометраж: ~N мин

[ХУК] первые 2 предложения...
[ТЕЗИС] первые 2 предложения...
[ТЕОРИЯ] краткое описание блока
[КЕЙС] краткое описание + ключевые цифры
[РЕЗЮМЕ] 3 пункта

Утверждаем? [OK / Изменить / Переделать]
```

**Если "Изменить"** — спросить что именно, поправить конкретный блок.
**Если "Переделать"** — спросить что не так, сгенерировать заново.
**Если "OK"** — перейти к Step 4.

### Step 4: Телесуфлёр

После утверждения — вывести чистый текст для телесуфлёра:

**Правила телесуфлёр-текста:**
- Убрать все тайм-коды и метки блоков ([ХУК], [ТЕЗИС] и т.д.)
- Убрать ремарки и комментарии
- Только чистый текст, который читается с суфлёра
- Разбить на абзацы по 2-3 предложения (удобно читать)
- Крупные смысловые блоки разделить пустой строкой
- Сохранить разговорный ритм: так, как говорят вслух, а не пишут

Вывести текст в чат — пользователь копирует его в приложение телесуфлёра.

### Step 5: Сохранить DOCX

**Всегда** после вывода телесуфлёр-текста — создать DOCX файл в `.tmp/teleprompter/`.

**Имя файла:** `YYYY-MM-DD-HHMM-Русский заголовок.docx`, где:
- `YYYY-MM-DD-HHMM` — дата и время отложенной публикации (`pub_date` + `pub_time` из Step 1)
- `Русский заголовок` — короткий человекочитаемый заголовок на русском языке, без транслита

Пример: `2026-06-25-1400-Топ 5 экономических инструментов.docx`

**Важно:** имя файла = время публикации. По нему `/publish-from-script` потом найдёт DOCX и поставит посты в очередь Google Sheets на это же время. Имя видеофайла в Google Drive должно быть таким же (без `.docx`, с `.mp4`).

**Правило русского имени:** не транслитерировать и не заменять пробелы дефисами. Сохранять русский заголовок в нормальном виде: `Топ 5 экономических инструментов`, `Как ИИ заменяет закупщиков`, `Бутылочное горлышко в производстве`. Удалять только символы, запрещённые Windows-путями: `< > : " / \ | ? *`. Если заголовок длиннее 80 символов, сократить смыслово на русском, не переходя на латиницу.

**DOCX — единый источник публикаций:** этот же файл без ручной чистки используется в `/publish-from-script` для WordPress-блога и Яндекс Дзен через RSS. Поэтому в теле DOCX не должно быть служебных строк для монтажа или подсказок вроде `ВИЗУАЛ НА ЭКРАН`, `СХЕМА НА ЭКРАН`, `КАРТИНКА НА ЭКРАН`. Любой такой текст попадёт в статью.

**Mandatory visual inserts:**
- If the script needs any table, chart, dashboard, scheme, matrix, KPI block, before/after comparison, funnel, process map, or numeric breakdown, generate it as an image with built-in GPT Image 2 (`gpt-image-2`) first and embed that image into the DOCX with `python-docx`.
- Do not require, check, or wait for `OPENAI_API_KEY` for these visuals. The model/tooling has built-in GPT Image 2 access for this workflow.
- Do not use local Pillow/HTML/SVG-generated substitutes as a fallback merely because `OPENAI_API_KEY` is absent. Only use a non-GPT-image fallback if the built-in image-generation tool itself fails or the user explicitly asks for a technical placeholder.
- Never leave analytical tables or visual structures as plaintext, Markdown tables, ASCII diagrams, pasted spreadsheet cells, or manually screenshot-needed content.
- Do not add visible sync markers or video-only captions in the DOCX body. The actual table/scheme/dashboard must be the embedded image placed between normal paragraphs. If the narration needs to introduce the visual, do it as natural article-safe prose, not as a production note.
- Store generated source images next to the DOCX under `.tmp/teleprompter/<docx-base>_assets/` and insert the same files into the DOCX, so `/render-docx-visuals` can extract them from `word/media/*`.
- Use a 16:9-safe business-slide layout, readable at 1920x1080: large labels, high contrast, no tiny text, no decorative stock imagery, no transcript text inside the visual.
- Before finishing, verify the DOCX ZIP contains `word/media/*` images for every embedded visual, that no Markdown/plaintext table remains in the body, and that `read_docx.py` output contains no service text such as `ВИЗУАЛ НА ЭКРАН`.

**Формат DOCX:**
- Шрифт: Arial 16pt (крупный, удобно читать с суфлёра)
- Межстрочный интервал: 1.4
- Заголовок: level 0, по центру
- **Под заголовком — строка `Публикация: YYYY-MM-DD HH:MM MSK`** (мелким серым шрифтом)
- Абзацы: 2-3 предложения, отступ после каждого 10pt
- Смысловые блоки разделены серым `— — —`
- Поля страницы: 2 см сверху/снизу, 2.5 см по бокам

Использовать `python-docx` через Bash. Не забывать `sys.stdout = io.TextIOWrapper(sys.stdout.buffer, encoding='utf-8')` для Windows.

### Step 6: Мини-гайд после DOCX

После успешного создания и проверки DOCX всегда вывести в финальном ответе короткий блок:

```
Дальше по SOP:
1. Записать raw-видео по DOCX.
2. Запустить /render-docx-visuals raw.mp4 --docx <docx_path>.
3. Проверить rendered MP4 в .tmp/rendered/YYYY-MM-DD-HHMM-Русский заголовок.mp4.
4. Загрузить rendered MP4 в Google Drive с тем же base-name, не raw.
5. После n8n upload взять YouTube URL из служебного уведомления.
6. Запустить /publish-from-script <youtube_url>.
```

## Примеры тем

Для понимания контекста — типичные темы канала Веселкова:
- ТОС (теория ограничений) в управлении запасами
- Как ИИ заменяет закупщиков — и как на этом заработать
- Бутылочное горлышко в производстве — где теряешь деньги
- Инвестиции вне бутылочного горлышка — почему это слив бюджета
- Пенсионер с опытом 20+ лет: алгоритм выхода на 350К/мес

## Чего НЕ делать

- Не генерировать thumbnail (это задача n8n)
- Не оставлять таблицы, схемы, дашборды, KPI-блоки или сравнения в текстовом формате: они всегда должны быть embedded images, сгенерированные через встроенный `gpt-image-2`
- Не требовать `OPENAI_API_KEY` для генерации visual inserts в DOCX
- Не добавлять в DOCX служебные строки вроде `ВИЗУАЛ НА ЭКРАН`, потому что DOCX без изменений идёт в WordPress и Яндекс Дзен
- Не завершать создание DOCX без мини-гайда следующих шагов SOP
- Не рассчитывать на ручные скриншоты после генерации DOCX
- Не публиковать сценарий никуда (это рабочий документ)
- Не предлагать "улучшить" стиль — стиль определён в tone-of-voice.md
- Не делать сценарий длиннее 12 минут без явной просьбы
- Не добавлять мотивационные блоки, психологию, "мышление успеха"
